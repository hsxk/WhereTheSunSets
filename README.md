# WhereTheSunSets

一个纯前端的日出、日落、月升、月落方位可视化工具。

项目最初是为了回答一个很具体的问题:

“我什么时候能在某个观察点，看到太阳正好落到某个目标物后面？”

现在它已经扩展成一个单文件静态页面工具，支持:

- 地理定位或地图点选位置
- 指定日期查看日出、日落、月升、月落时间与方位角
- 地图方向引导线
- 天穹轨迹图
- 金色时刻 / 蓝色时刻提示
- 摄影用目标点对日分析
- 多语言界面
- URL 分享当前经纬度与日期

仓库当前只有一个核心文件:

- [index.html](/Users/haokexin/service/WhereTheSunSets/index.html)

这意味着它部署非常简单，但也意味着样式、结构、业务逻辑和多语言文案都集中在同一个文件中。

## 在线能力概览

页面主要提供以下能力:

1. 通过浏览器定位或地图点击确定观察点
2. 查看当天或任意日期的太阳/月亮关键事件
3. 在地图上绘制日出、日落、月升、月落的方向线
4. 在天穹图中查看太阳和月亮当天的轨迹
5. 输入或地图点选目标点，分析未来一段时间内是否出现“日出/日落穿过目标方向”的机会
6. 通过 URL hash 分享当前日期和位置

## 技术实现

项目没有构建系统，没有打包步骤，也没有后端。

核心依赖通过 CDN 直接加载:

- [SunCalc](https://github.com/mourner/suncalc)
- [Leaflet](https://leafletjs.com/)
- OpenStreetMap 瓦片

页面运行方式:

- HTML: 页面结构
- 内联 CSS: 视觉样式
- 内联 JavaScript: 状态管理、天文计算、地图交互、多语言逻辑

## 快速开始

### 方式 1: 直接打开

可以直接双击打开 [index.html](/Users/haokexin/service/WhereTheSunSets/index.html)。

但需要注意:

- 某些浏览器下，地理定位和剪贴板能力在 `file://` 协议下可能受限
- 如果你要完整测试定位、分享、跨设备表现，建议走本地 HTTP 服务

### 方式 2: 本地静态服务

在项目目录下启动任意静态文件服务器，例如:

```bash
python3 -m http.server 8080
```

然后访问:

```text
http://localhost:8080
```

## 交互说明

### 1. 选择位置

- 点击“Locate Me / 一键定位”使用浏览器定位
- 或直接点击地图选定观察点
- 位置会保存到 `localStorage`

### 2. 切换日期

- 使用日期输入框
- 使用左右按钮逐天切换
- 使用左右方向键切换
- 在移动端可以左右滑动主内容区切换日期
- 使用滑块快速跳到一年中的某一天

### 3. 查看核心数据

页面右侧会显示:

- 日出时间和方向
- 日落时间和方向
- 月升时间和方向
- 月落时间和方向
- 太阳正午
- 白昼时长
- 月面照明比例
- 月相

### 4. 查看地图引导

地图上会绘制:

- 日出方向线
- 日落方向线
- 月升方向线
- 月落方向线
- 观察点到目标点的辅助连线

### 5. 使用摄影分析

展开“高级功能 / Advanced”区域后:

1. 先设置观察点
2. 再设置目标点
3. 工具会计算目标方位角、距离，以及未来扫描范围内可能的“日出过点 / 日落过点”日期

## URL 与状态持久化

项目目前使用两种方式保存状态:

### localStorage

用于保存:

- 上次使用的语言
- 上次选择的观察点坐标

键名:

- `sm.lang`
- `sm.lat`
- `sm.lng`

### URL hash

用于分享:

- `lat`
- `lng`
- `date`

示例:

```text
#lat=35.3606&lng=138.7274&date=2026-11-13
```

打开分享链接后，页面会优先读取 hash 中的状态。

## 我建议优先优化的“操作性”

下面这些不是“代码不能运行”的问题，而是更偏向产品操作体验和后续维护效率的优化点。

### 1. 高级模式的地图点选流程不够显式

当前高级模式下，用户需要记住“第一次点地图选观察点，第二次点地图选目标点”，但页面几乎没有持续性的步骤提示。

对应逻辑可见:

- [index.html](/Users/haokexin/service/WhereTheSunSets/index.html#L999)

建议:

- 在高级面板里增加明显的步骤状态提示
- 当前处于“请选择观察点”还是“请选择目标点”时，给出醒目文案
- 给目标点和观察点不同的 marker 样式与图例
- 增加“重新选择观察点 / 重新选择目标点”按钮

这是我认为最值得优先做的一项，因为它直接影响用户第一次是否能顺利用起来。

### 2. `clearAll()` 会清掉地图标记，但不会清空当前观察点状态

当前 `clearAll()` 只移除了 marker、guide layer 和目标点输入框，但没有重置全局 `lat/lng`，也没有清空观察点输入框。

对应逻辑可见:

- [index.html](/Users/haokexin/service/WhereTheSunSets/index.html#L1038)

这会导致界面看起来像“已经清空”，但内部状态仍可能保留旧观察点，后续分享、渲染、分析容易让用户困惑。

建议:

- 明确区分“仅清目标点”和“重置全部”
- 如果保留现有观察点，就把按钮文案改成更准确的语义
- 如果按钮语义就是清空全部，那就同步清空 `lat/lng`、`obsLat/obsLng`、本地存储和状态栏

### 3. 首次进入就自动定位，容易触发突兀权限请求

初始化时如果没有 hash 和本地缓存，会直接调用 `doLocate()`。

对应逻辑:

- [index.html](/Users/haokexin/service/WhereTheSunSets/index.html#L1321)

建议:

- 首屏先展示引导，不自动弹定位权限
- 让用户主动点击“定位我”再请求权限
- 如果想保留自动定位，可以只在移动端或 PWA 场景开启

这样能减少首次拒绝权限后带来的挫败感，也更符合很多浏览器的权限预期。

### 4. URL hash 只保存了观察点与日期，没有保存高级分析上下文

当前分享链接只包含:

- `lat`
- `lng`
- `date`

对应逻辑:

- [index.html](/Users/haokexin/service/WhereTheSunSets/index.html#L1256)

如果用户在做“目标对日分析”，分享出去后目标点、海拔、是否展开高级面板都会丢失。

建议:

- 追加 `tgtLat`、`tgtLng`、`obsElev`、`tgtElev`
- 可选保存 `adv=open`
- 这样分享链接可以真正复现摄影分析场景

### 5. 多语言文本全部内联，维护成本会越来越高

当前六种语言全部写在同一个对象中，和页面逻辑耦合在一起。

对应区域:

- [index.html](/Users/haokexin/service/WhereTheSunSets/index.html#L408)

建议:

- 至少把 `I18N` 拆到单独的 `i18n.js` 或 `locales/*.json`
- 为缺失翻译增加检测脚本或运行时告警
- 给文案 key 建立统一命名规范

这项优化对用户的直接感知没那么强，但对后续继续扩展语言和文案会很重要。

### 6. 单文件结构已经接近“值得拆分”的阈值

当前样式、结构、业务、地图、天穹绘图、分享、i18n 都在同一个文件中，阅读和修改成本较高。

建议的最低拆分方案:

- `index.html`: 页面骨架
- `styles.css`: 样式
- `app.js`: 启动与状态管理
- `astro.js`: SunCalc 包装与方位计算
- `map.js`: 地图与 marker 交互
- `i18n.js`: 文案与语言切换

这不是为了“工程化而工程化”，而是为了让你后面继续迭代时不至于每次都在一个 1300+ 行文件里找逻辑。

### 7. 高级分析的计算范围和性能策略可以做成可配置

`findPass()` 默认扫描未来 550 天。

对应逻辑:

- [index.html](/Users/haokexin/service/WhereTheSunSets/index.html#L1077)

这对单次使用还可以，但如果以后增加更多事件类型、批量目标或更复杂地形校正，性能会比较紧。

建议:

- 将扫描范围做成常量或 UI 配置
- 分析中增加 loading 状态
- 后续可以考虑 `requestIdleCallback` 或 Web Worker

### 8. 目前没有显式的开发说明、限制说明和验证方式

这项不是界面问题，但属于“项目操作性”的一部分。

例如:

- 为什么建议用本地 HTTP 服务而不是直接双击打开

- `file://` 场景下可能哪些功能失效
- 如何验证地理定位、分享链接、多语言和高级分析

这也是这次 README 补充的重点。

## 已知限制

- 天文计算基于 SunCalc，属于近似/参考用途
- 未接入地形遮挡、DEM、高山轮廓或建筑物遮挡计算
- 目标点分析基于方位角相交，不等于严格可见性分析
- 地图底图依赖 OpenStreetMap 瓦片服务
- 外部 CDN 不可用时，页面会失去对应功能
- 在高纬度地区，某些日期可能没有日出或日落

## 后续可选演进方向

如果你准备继续把它做成更完整的工具，我建议可以按这个顺序推进:

1. 先补操作引导和状态语义
2. 再把分享状态扩充完整

## 手动测试建议

每次改动后至少验证这些场景:

1. 首次打开页面，无缓存时是否能正常进入
2. 点击定位后是否能正确更新地图、卡片和天穹图
3. 点击地图后是否能更新观察点
4. 左右切换日期、滑块拖动是否流畅
5. 分享链接在新窗口打开后是否能恢复状态
6. 切换语言后主要按钮和状态文案是否同步更新
7. 高级分析中“观察点 -> 目标点”两步选择是否清晰

## 维护建议

如果你准备继续维护这个项目，建议下一步最小动作不是重构全部代码，而是先做这三件事:

1. 明确“清除”按钮的语义并修正状态同步
2. 给高级分析加一步一步的引导提示
3. 把 README 和部署说明固定下来

这样投入小，但收益很高。
